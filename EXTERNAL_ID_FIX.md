# Исправление UNIQUE constraint на external_id - 01.05.2026

## Проблема

При импорте заказов от нового менеджера возникала ошибка:

```
UNIQUE constraint failed: clients.external_id
[SQL: INSERT INTO clients (external_id, phone, name, ...) VALUES (?, ?, ?, ...)]
[parameters: ('CLIENT-2', '+79281551515', 'имя', ...)]
```

## Причина

### Архитектурная проблема:

В системе есть два источника создания клиентов:

1. **Из файла orders_*.json** - когда заказ ссылается на клиента
2. **Из файла clients_*.json** - когда менеджер явно создает/обновляет клиента

**Проблема:** Разные менеджеры на своих устройствах могут создавать клиентов с одинаковыми `external_id`:
- Менеджер 1232 создает клиента `CLIENT-2`
- Менеджер 5678 тоже создает клиента `CLIENT-2`

При импорте в центральную систему возникает конфликт, так как `external_id` имеет UNIQUE constraint.

### Правильная логика:

**Уникальный идентификатор клиента = телефон** (не external_id)

- Один телефон = один клиент в центральной системе
- `external_id` - это локальный ID на устройстве менеджера
- При импорте нужно искать клиента по телефону, а не по external_id

## Решение

### 1. Метод _get_or_create_client (строки 269-287)

**Было:**
```python
def _get_or_create_client(self, session, order_data: Dict) -> Client:
    phone = order_data["client_phone"]
    client = session.query(Client).filter_by(phone=phone).first()

    if not client:
        client = Client(
            external_id=order_data.get("client_id"),  # ❌ Конфликт!
            phone=phone,
            ...
        )
```

**Стало:**
```python
def _get_or_create_client(self, session, order_data: Dict) -> Client:
    phone = order_data["client_phone"]
    client = session.query(Client).filter_by(phone=phone).first()

    if not client:
        # Не устанавливаем external_id при создании из заказа,
        # так как он может конфликтовать с клиентами от других менеджеров
        client = Client(
            external_id=None,  # ✅ Будет установлен при импорте из clients_*.json
            phone=phone,
            ...
        )
```

### 2. Метод _process_clients (строки 373-409)

**Было:**
```python
if client:
    # Обновляем существующего клиента
    client.name = client_data.get("name", client.name)
    ...
else:
    # Создаем нового клиента
    client = Client(
        external_id=client_data["id"],  # ❌ Конфликт между менеджерами!
        phone=phone,
        ...
    )
```

**Стало:**
```python
if client:
    # Обновляем существующего клиента
    client.name = client_data.get("name", client.name)
    ...
    # Обновляем external_id только если он не установлен
    if not client.external_id:
        client.external_id = f"{manager.pin_code}-{client_data['id']}"  # ✅ Уникальный!
else:
    # Создаем нового клиента
    # Делаем external_id уникальным: PIN-CLIENT_ID
    client = Client(
        external_id=f"{manager.pin_code}-{client_data['id']}",  # ✅ Уникальный!
        phone=phone,
        ...
    )
```

## Как это работает теперь

### Сценарий 1: Импорт заказа с новым клиентом

1. Приходит файл `orders_1234567890.json` от менеджера 1232
2. Заказ ссылается на клиента с телефоном `+79281551515`
3. `_get_or_create_client()` ищет клиента по телефону
4. Если не найден → создает с `external_id=None`
5. Заказ успешно импортируется

### Сценарий 2: Импорт данных клиента

1. Приходит файл `clients_1234567890.json` от менеджера 1232
2. Клиент имеет `id="CLIENT-2"` и телефон `+79281551515`
3. `_process_clients()` ищет клиента по телефону
4. Если найден → обновляет данные и устанавливает `external_id="1232-CLIENT-2"`
5. Если не найден → создает с `external_id="1232-CLIENT-2"`

### Сценарий 3: Два менеджера с одинаковым CLIENT-2

**Менеджер 1232:**
- Создает клиента `CLIENT-2` с телефоном `+79281551515`
- В центральной системе: `external_id="1232-CLIENT-2"`

**Менеджер 5678:**
- Создает клиента `CLIENT-2` с телефоном `+79287654321` (другой телефон)
- В центральной системе: `external_id="5678-CLIENT-2"`

✅ Нет конфликта! Разные телефоны = разные клиенты

**Менеджер 5678:**
- Создает клиента `CLIENT-3` с телефоном `+79281551515` (тот же телефон!)
- В центральной системе: находит существующего клиента по телефону
- Обновляет данные, но НЕ меняет `external_id` (остается `1232-CLIENT-2`)

✅ Один телефон = один клиент в системе

## Преимущества решения

✅ **Нет конфликтов** - каждый менеджер имеет свой префикс в external_id
✅ **Один телефон = один клиент** - правильная бизнес-логика
✅ **Обратная совместимость** - старые клиенты без external_id продолжают работать
✅ **Трассируемость** - по external_id видно, какой менеджер создал клиента

## Альтернативные решения (не выбраны)

### Вариант 1: Убрать UNIQUE constraint с external_id
❌ Плохо: потеряем уникальность, возможны дубликаты

### Вариант 2: Использовать составной ключ (manager_id, external_id)
❌ Сложно: требует изменения схемы БД и миграции

### Вариант 3: Генерировать UUID для external_id
❌ Плохо: теряется связь с устройством, сложнее отладка
